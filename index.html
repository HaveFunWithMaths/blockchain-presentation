Act 1 · The Problem

What people think it is

Where it came from

Satoshi Nakamoto, 2008

The invention wasn't digital money.
It was a shared record no one controls and no one can secretly rewrite.

Bitcoin: A Peer-to-Peer Electronic Cash System
Satoshi Nakamoto · satoshin@gmx.com · www.bitcoin.org
Abstract. A purely peer-to-peer version of electronic cash would allow online payments to be sent directly from one party to another without going through a financial institution…
People tried digital money for decades and it kept failing — the hard part was never the coins, it was agreeing on a record with no trusted keeper. Satoshi's 2008 paper cracked that. Key framing: **blockchain = the mechanism; Bitcoin = the first application.** CERN parallel: the web was invented to share physics papers, then ate the world — same story here, built for one problem, escaped into many.

Act 2 · The Engineering

A shared, distributed ledger

▤▤▤ ▤▤▤ ▤▤▤

One ledger, mirrored on every computer. No owner.

  • How does everyone agree what's true?
  • What stops someone editing their own copy?
Imagine what Alice writes instantly appears on Bob's copy — and nobody owns it; everyone holds a full copy. No middleman; the same idea works for 2 people, 10,000, or the whole public. But going owner-less raises the two questions the rest of Act 2 answers: **(1) how does everyone agree what's true?** and **(2) what stops someone quietly editing their own copy?**

Here's the elephant in the room. Every copy of the ledger lives on somebody's machine. What stops a bad actor from quietly rewriting an old entry to pay themselves? Naively — **nothing.** This is *the* problem all the remaining engineering exists to kill. Cue the reveal: hashing, chaining, and consensus. Sit in the discomfort for a beat before the next slide.

Hashing, explained

Any data → a fixed-size fingerprint.

Deterministicsame input → same output
Avalanche1 char changes → totally different
One-waycan't reverse it

Hashing ≠ encryption — encryption is reversible, hashing is not.

A hash maps any input to a fixed-size fingerprint. Three properties: same input → same output (deterministic); change one character → a completely different output (avalanche); and you can't run it backwards (one-way). Engineers know these — SHA-256, Git SHAs. Say the common confusion out loud and kill it: **hashing is not encryption.** Encryption is reversible by design (you decrypt); hashing is a one-way fingerprint — there is no "un-hash."

Stage A — MiniHash (see every step) → Stage B — SHA-256 (the real one)

Demo 1 mounts here — MiniHash → SHA-256, live-editable
Stage A — the toy MiniHash: change one letter and watch the running value diverge; make the point that the fingerprint is *entirely determined by the data* and hypersensitive to it. Then reveal Stage B: "here's the industrial-strength version — same idea, same avalanche, just uncrackable." The toy demystifies *how*; SHA-256 makes it real. Don't rush this — it's one of the two demos the whole talk rests on. (Editing one character must visibly change the whole fingerprint.)

Why hash — chaining them

BLOCK N-1
data: …
hash: a3f9…
⛓
BLOCK N
prevHash: a3f9…
hash: 7c1e…

Each block embeds the previous block's hash → blocks are cryptographically welded in order.

A hash is a fingerprint/signature of the data: store it now, re-hash later, compare — any mismatch means tampering. The trick that turns a pile of blocks into a *chain*: each block embeds the previous block's hash as its `prevHash`, so they're cryptographically welded in order. Change any earlier block and its hash changes, so the next block's stored `prevHash` no longer matches. (Optional one-liner: the transactions *inside* a block are also hashed together into a Merkle tree — but we won't go deeper today.)

🔴 Live: digital signatures

Tamper-evidence proves the data wasn't changed. Signatures prove who authorized it.

Signatures demo mounts here — sign as Alice, verify with her public key

🔒 Private key

Signs. Never shared. Only Alice holds it.

🔑 Public key

Verifies. Shared with everyone. Proves it was Alice.

The gap this fills: the tamper→domino proved the ledger can't be secretly *altered* (integrity) — but not that Alice actually *authorized* her transaction (authenticity). Naively, anyone could type "Alice → Bob : 50" into a block. Public-key cryptography closes it. Alice has a **key pair**: a **private key** she keeps secret and a **public key** she shares freely. To send, she **signs** the transaction with her private key — the signature is a number derived from both the message *and* her secret. Anyone with her public key can **verify** that signature matches the message: proof it came from her and wasn't changed. Live: hit **Sign as Alice**, then edit one character (or "Forge a char") — the same signature stops verifying (✗ forged). The punch: an attacker can copy her *public* key but still can't forge a signature without the *private* key. **Verifying is public; signing is private.** This is how every real transaction is authorized — the sender signs, every node verifies before accepting. (Still true: hashing ≠ encryption — signing uses the key pair, it doesn't reversibly encrypt the message.) Keep it to ~2 min; the live forge attempt is the whole point.

🔴 Live: tamper → domino

Edit a past transaction; watch every downstream block break.

Demo 2 mounts here — ~4 linked blocks, edit → cascade, fix / reset
This is the emotional peak — give it room. Edit the data in block 2: its hash changes → block 3's stored `prevHash` no longer matches → the break cascades red down the chain. Say it plainly: **tampering isn't prevented — it's made instantly visible** (tamper-evident, not tamper-proof). Then the hook into consensus: "to hide this you'd have to recompute *every* block after it, on *every* copy, faster than everyone else — which is exactly what consensus makes absurdly hard." Use "Fix chain" to show how much work hiding it would take.

Consensus & Proof-of-Work

▤▤▤ ▤▤▤

Longest valid chain wins → honest majority converges.

To add a block, find a nonce so the hash starts with N zeros:

0000a91f…c3  ← hard to find, trivial to verify

51% attack: you'd need most of the world's compute to rewrite history.

This slide gets real depth. • **The problem:** with thousands of copies — and sometimes two valid blocks appearing at once — which chain is *the* truth? Consensus = the network's rules for agreeing on one canonical chain. • **Proof-of-Work:** to add a block you must find a **nonce** that makes the block's hash start with N leading zeros. Hard to find (brute force), trivial to verify. Writing history is made expensive. • **Why tampering dies here (callback to the domino):** rewriting block 2 means redoing the Proof-of-Work for block 2 *and every block after it*, then out-racing the entire honest network that keeps extending the real chain. Economically irrational. • **Longest/heaviest valid chain wins** → the honest majority converges. • **51% attack:** you'd need a majority of the world's compute to rewrite history — the security model is "cheating costs more than it's worth." • **Hand off to the next slide:** there's a second way to make writing expensive that doesn't burn electricity — Proof-of-Stake — which is its own slide next. Don't dive into it here.

Proof-of-Work vs Proof-of-Stake

Two ways to make writing history expensive — so honesty is the cheapest strategy.

Proof-of-Work

  • Burn compute / electricity to earn the right to add a block
  • Security = energy already spent
  • Cost: enormous power draw — a small country's worth
  • Bitcoin

Proof-of-Stake

  • Post collateral (stake); cheat → it's slashed (burned)
  • Security = money at risk
  • ~99.9% less energy — no mining race
  • Ethereum (switched in 2022)

Same goal — attacking must cost more than it earns. PoW pays in energy; PoS pays in stake.

Expand the one-liner I deferred on the last slide. Both schemes answer the same question — "how do we make *writing* history expensive so the honest chain always wins?" — with different currencies. • **Proof-of-Work:** to add a block you burn real compute (and thus electricity) searching for the nonce. Your stake in honesty is the energy you've already spent; rewriting history means re-spending it faster than everyone else. This is where the **energy debate** lives: Bitcoin's network draws on the order of a mid-size country's electricity, which is the single most common criticism — sometimes countered with "much of it uses stranded/renewable power," but it's genuinely large. • **Proof-of-Stake:** instead of burning energy, validators lock up **collateral** — a stake. Misbehave and the protocol **slashes** (destroys) it. Security comes from money-at-risk, not power-spent, so it uses ~99.9% less energy. Ethereum switched from PoW to PoS in 2022 (**"The Merge"**), cutting its energy use dramatically. • **The trade-off framing to leave them with:** PoW ties security to a physical, external cost (electricity); PoS ties it to an internal, economic one (staked capital). Critics of PoS worry it can favor those who already hold the most coins ("rich get richer"); defenders note the collateral makes attacks self-punishing. Don't adjudicate — just show both are the same idea in different currency. Keep to ~1.5 min — it's a frequent Q&A magnet, so having it as a slide earns its place.

Act 3 · Why It Matters

Trust the data,
not the people.

This is the real invention, stated in one line. You no longer trust Alice, Bob, or the bank — you trust the shared record and the math that makes it tamper-evident. Trust moves from individuals and institutions *to the system itself*, even with no central authority. That's the whole magic trick, decoded — everything in Act 2 existed to earn this sentence.

Beyond money

A ledger needn't hold money — anything you want an agreed, tamper-evident history of:

Supply chains Property titles Medical records Credentials
The ledger doesn't care what the entries mean. Anything you want an agreed, tamper-evident history of fits. Use the hospital-records example: a patient's history that multiple providers can read and append to, where no single party can silently rewrite the past. (If running long, this slide merges with slide 16.)

Public vs permissioned

Public

  • Anyone can join
  • Anonymous, open read/write
  • Bitcoin, Ethereum

Permissioned

  • Known, authenticated members
  • Unique IDs, role-based access
  • Business networks
Blockchains are open by default — but businesses often want control. Permissioned chains have authenticated participants with unique IDs and differentiated (role-based) access — same tamper-evident ledger, but you know exactly who everyone is. Example: a hospital network sharing records across providers securely and consistently, where membership is controlled but no single provider owns the record.

Smart contracts

Store not just data — a small program that self-executes when conditions are met.

when (payment received)
    transfer(ownership) → buyer   // no intermediary, runs the same every time
Instead of storing just data, store a small program — an agreement written in code — that runs automatically when its conditions are met. Classic example: "payment received → ownership transfers," with no intermediary to enforce it. It's deterministic: it always runs the same way for everyone. Ethereum is the usual home for these. Give one crisp example and stop — the *why it costs money* beat is its own slide next.

Gas: why transactions cost money

Running that code isn't free — every operation has a price, paid as a fee ("gas") to whoever adds your block.

Pays for workcompensates validators + funds security
Stops spaman infinite loop would cost infinite money → it halts
Prices prioritybid more → mined sooner; busy network → higher fees

More computation = more gas. A simple transfer is cheap; a complex contract costs more.

Ties straight to the previous slide. If a smart contract is a program the whole network runs, someone has to pay for that computation — and it has to be metered, or a single bad contract could run forever and freeze the chain. That's **gas**: every operation has a small price, and you attach a **fee** to your transaction to cover it, paid to whoever adds your block. • **Pays for the work + security:** the fee compensates the miner / validator and, with the block reward, is what funds the honest majority we built in Act 2. • **Spam / halting protection:** because every step costs money, an infinite loop simply runs out of gas and stops — you can't DoS the network for free. This is the real reason gas exists, not just revenue. • **A live fee market:** gas is priced by supply and demand — bid more to get mined sooner; when the network is busy, fees spike (the famous "Ethereum gas fees"). More computation = more gas, so a plain transfer is cheap and a complex contract is dear. Keep it to ~1 min — the takeaway is just "computation is metered and paid for; that's what fees are." Ethereum is where the term lives.

Have a trusted central party?
Use a database.

Simpler · faster · cheaper. Blockchain earns its overhead only when participants don't trust each other and there's no central controller.

The credibility slide — say this out loud and the rest of the talk becomes believable. If you already have a trusted central party, a normal database is simpler, faster, and cheaper — full stop. Blockchain earns its considerable overhead **only** when participants *don't* trust each other *and* you want *no* central controller. Decoding the hype — telling people when *not* to use it — is literally the point of this series.
